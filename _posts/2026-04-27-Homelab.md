# Why?

Buying a new phone comes with some benefits, with my most recent phone including a year's trial of Google Gemini Pro, and a large amount of cloud storage. I filled this space very quickly, and then because ignorance is bliss, I was suddenly faced with a nice quote for continuing the subscription. No problem, let's just downgrade!

But no, my storage had exceeded the lower tiers and therefore would be deleted after x days. Fair enough, storage isn't cheap at the moment...

Annoyed by my oversight, and using some nerd-maths, I decided that building a dedicated home server was the most efficient and cheapest way of not paying a small monthly fee. An upside of this as well is local hosting, and therefore more privacy and learning opportunities. 

# What?

And so it began. I actually started out a few years ago, beginning with a Raspberry Pi 4b which I ran the fantastic Home Assistant on. This started as a simple few automations, but quickly grew to being something that we couldn't live without. For the new iteration of the homelab however, we needed something more powerful.

I ended up buying a used Lenovo ThinkCentre Tiny - a small form factor machine which is one of many that are perfect for this kind of application. My specific model is:
    
* Lenovo ThinkCentre Tiny M700
* Intel i5-6400
* 8 GB RAM
* 240 GB M2 SSD

For what I'm planning to run now and in the future, this is a great machine. It cost me just over £100 and as it's used, it's gets some environmental points as well. 

As this machine would live in my office and serve as the upstairs networking hub, I wanted to go for the mini-rack look, so purchased a Tecmojo 10in 6U rack - making the rack below!

![My Homelab]({{ 'images/homelab_cropped.jpeg' | relative_url }})

It might not be the most advanced or most beautiful build ever (I'm no Jeff Geerling), but it does lots of what I need. From top to bottom we have 1x RPI 4B, 12 port Cat6E patch panel, Netgear network switch, 2x hotswap drive bay with 1x 1 TB SSD and 1x 2 TB HDD (more on those later...) and finally the ThinkCentre in a GeekPi minipc shelf. Next to it is one of our home mesh nodes, and overall the rack leaves lots of room for expansion and reconfiguration in the future...

# How?

Now we have things mounted nicely, it was time to think about how to install the individual services I required. It also importantly needed to start as something that could over time because a local cloud for the household, and therefore requires some WAF (Wife Acceptance Factor). The key for this was using tools that were easy to access, otherwise it wouldn't be used.

For the main server which I've christened `srv001`, I decided on Ubuntu 24.02 Server (headless). I chose headless because I'm still relatively new to Linux in general, and have a somewhat established version of Ubuntu on my WSL2 install on my main PC. Installing this was easy, creating a bootable USB and removing the version of MS Windows and associated partitions on the internal M.2 drive. I configured the main machine using SSH.

For the "interactive" layer that's needed for people who don't like CLIs, I went for CasaOS. After researching things like Proxmox (and other hypervisors) I decided that for the family use that I envisioned, a easier interface might be the best option long-term. CasaOS provides essentially a web interface for the server, users, and an overview of services on the machine along with a familiar file explorer. Installing was easy, running the following code in SSH: 

    wget -qO- https://get.casaos.io | sudo bash

Once downloaded, you are met with a simple walkthrough which I won't go into massively here, as there are far better resources available a quick search away.

### Containerisation

A big value of these kind of builds is the ability to use systems like `Docker` to install containers - essentially tiny virtual machines that can be isolated from each other or given free reign to communicate. The first install was simple: Home Assistant.

As I've previously used the Raspberry Pi for this, it was an incredibly simple transition. I installed HomeAssistant from Docker using the SSH command `docker pull homeassistant/home-assistant`, and then backed up my previous server from a download. I was very suprised that everything worked immediately - I've recently had some issues with connecting my car to HA so was expecting lots of uninstalling and reinstalling, but everything went according to plan.

Another must-have for the house was a local media service. I chose Jellyfin - a relatively simple media service that would provide all of the metadata for the media we pointed it to. This service was also the key for having the external storage drives, which while not ideally set up at the moment, would provide an easy way to move media from our personal machines onto something that is easily accessible from every TV in the house.

For Jellyfin, I tried out the CasaOS "App Store" - a curated collection of Docker Images which can be customised further either through the UI or via `docker-compose.yml` files. After installing the image, I then pointed the compose files to my external storage, and started populating with media. I set up a dedicated account for the TV, logged in, and as easy as that we had our own media server!

Docker Compose is a powerful tool, so if doing something similar I would highly recommend learning - for instance you _could_ put certain P2P containers within their own VPN if required - that have built in killswitches to kill the container should the VPN connection drop. 

I have also installed an `immich` container; which acts as a local hosted version of Google Photos. Photos of my dog after all make up most of the storage that I use in Google Photos, so hosting at home makes sense. I still need to follow some backup rules (3,2,1) and I would prefer to move to some sort of RAID setup for the drives, but we now have a solid backup which can be used alongside some of the paid cloud services.

### Backup and redundancy

The Raspberry Pi has a purpose in the new rack, and runs (at the moment) a dedicated Gitea instance. Gitea is a local version of Github, which is perfect for source controlling the Docker Compose files without exposing them (and the secrets which are hardcoded into the files) to the cloud. I think that given the [extended use of data within Github to aid AI training](https://github.blog/changelog/2026-03-25-updates-to-our-privacy-statement-and-terms-of-service-how-we-use-your-data/#ai-model-training-what-you-need-to-know), self hosting items such as these is extra important. 

With the Raspberry Pi running on an SD card, I wanted to ensure that there were also backups of the Gitea database and associated data, which I've automated to backup to the main PC nightly using a `cron` job. I am planning to write another blog about how I have achieved the backup automation that copies both key parts of the puzzle to the other machine, so stay tuned!
